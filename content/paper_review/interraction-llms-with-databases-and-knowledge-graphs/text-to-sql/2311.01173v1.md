Les générateurs de Text-to-SQL existants nécessitent que tout le schéma soit codé avec le texte de l’utilisateur. Ceci est coûteux ou peu pratique pour les grandes bases de données avec des dizaines de milliers de colonnes. Les techniques de retrait densément standard sont insuffisantes pour le sous-ensemble de schéma d’une grande base de données structurée, où la sémantique correcte du retrait nécessite que nous classions les _ensembles_ d’éléments de schéma plutôt que des éléments individuels. En réponse à cela, les auteurs de cette étude proposent un processus en deux étapes pour une couverture efficace pendant la récupération. 

D'abord, ils demandent à un modèle de langage (LLM) d'halluciner un schéma de base de données minimal jugé suffisant pour répondre à la requête. Ils utilisent le schéma halluciné pour extraire un sous-ensemble du véritable schéma, en composant les résultats de _plusieurs_ récupérations denses. Remarquablement, l'hallucination -- généralement considérée comme une nuisance -- s'avère en fait utile en tant que mécanisme de pont. Comme il n'existe pas de bancs d'essai existants pour le sous-ensemble de schéma sur de grandes bases de données, les auteurs introduisent trois bancs d'essai pour évaluer les performances de leur méthode.

Les auteurs ont utilisé GPT-3 pour halluciner un schéma minimal à partir d'une requête utilisateur. Ils ont ensuite utilisé ces schémas hallucinés pour rechercher des sous-ensembles pertinents du véritable schéma. Cependant, plutôt que de simplement chercher les éléments du schéma individuellement, ils ont évalué les _ensembles_ d'éléments, maximisant la couverture et récompensant la connectivité entre les éléments. Pour évaluer leur méthode, les auteurs ont développé trois nouvelles références, y compris une véritable base de données contenant près de 18000 éléments de schéma. La méthode proposée a conduit à un rappel nettement supérieur, ce qui se traduit par une plus grande précision de la génération de Text-to-SQL.

Par ailleurs, les tests menés par les auteurs sur deux différents ensembles de données ont montré que leur méthode, basée sur l'hallucination de schéma, améliore nettement les performances par rapport aux méthodes de récupération dense ordinaire.

Les auteurs ont également constaté que l'amélioration du rappel des éléments du schéma se traduisait par une plus grande précision de la génération de SQL par les systèmes de conversion Text-to-SQL. Ils ont également examiné l'impact de variations dans la méthode CRUSH, tels que le nombre d'exemples dans le contexte et le type de sollicitation pour l'hallucination, et ont constaté que même avec des variations, la méthode CRUSH surpassait les méthodes alternatives et était robuste face à ces variations.

Il y a deux limitations à CRUSH qui pourraient être utiles à aborder dans les travaux futurs. D'abord, à l'heure actuelle, l'hallucination est presque complètement non guidée par le schéma de la base de données client. Il serait intéressant d'explorer si le schéma de la base de données client peut être comprimé dans un texte d'invocation très petit pour donner une certaine guidance limitée à l'hallucinateur de schéma LLM. À l'heure actuelle, les poids du lien \(e(d,d^{\prime})\) entre les éléments de schéma \(d,d^{\prime}\) sont câblés en dur; il pourrait être utile d'étendre l'optimisation de l'apprentissage pour ajuster ces poids. 

Enfin, l'approche CRUSH pourrait également être adaptée à d'autres applications de récupération d'information structurée, au-delà de la conversion Text-to-SQL. 

Les auteurs remercient Microsoft pour avoir soutenu l'accès à l'API Azure OpenAI via l'initiative de recherche académique sur les modèles fondamentaux d'Accélérer.
