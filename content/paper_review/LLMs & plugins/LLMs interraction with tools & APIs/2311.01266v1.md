Comme rapporté récemment, le papier de recherche "Let's Discover More API Relations: A Large Language Model-based AI Chain for Unsupervised API Relation Inference", souligne l'importance des relations API pour les ingénieurs logiciels, les bibliothèques et les framework. Connaître les relations entre les API est crucial pour décider laquelle est la mieux adaptée pour une tâche donnée.

Ces relations sont décrites dans le texte, comme les spécifications de l'API, les tutoriels et les forums de questions-réponses. Cependant, en raison de la nature non structurée du texte, les riches connaissances sur les relations d'API dans les textes ne sont pas facilement accessibles pour la recommandation d'API, l'analyse des mauvais usages et la génération de code. Par conséquent, une façon efficace d'améliorer l'accessibilité des connaissances sur les relations d'API est de les extraire à partir de textes et de les représenter explicitement sous la forme d'informations structurées dans un graphe de connaissances.

Les méthodes de recherche pour extraire les relations API à partir de textes non structurés, se basent principalement sur des modèles de phrases élaborés manuellement ou sur des modèles pré-entrainés avec des phrases étiquetées. Cependant, ces approches présentent deux limites : 

D'une part, elles dépendent fortement du corpus de textes d'API collecté, qui ne peut pas couvrir de manière adéquate les diverses relations d'API. De plus, les méthodes actuelles n'extraient que les relations d'API explicitement mentionnées dans le texte donné, mais ne peuvent pas faire d'inférence de relations d'API au-delà du corpus de texte fourni.

D'autre part, l'efficacité des méthodes d'extraction de relations est affectée par les caractéristiques du texte en entrée, telles que la complexité syntaxique et la richesse sémantique. Les textes simples et facile à gérer peuvent ne pas capturer les relations d'API diverses, tandis que les textes complexes ayant une grande complexité syntaxique, comprennent de multiples clauses et références de pronoms, posant des défis pour l'extraction des relations.

Pour pallier ces deux limites, l'équipe de recherche propose une nouvelle méthode qui tire parti du modèle de langage de grande taille (LLM), par exemple GPT 3.5, comme une base de connaissances neurale pour inférer les relations API. Ces LLMs sont pré-entraînés sur le corpus du web entier, appelé les modèles de fondation. L'approche de l'équipe ne suppose pas que les relations d'API sont explicitement mentionnées dans un morceau de texte. Au lieu de cela, ils consultent le LLM pour la connaissance (par exemple, l'usage, les caractéristiques, la performance) des deux API séparément et ensuite font l'inférence des relations d'API sur la base de cette connaissance.

En exploitant la possibilité d'apprentissage en contexte du LLM, leur approche est complètement non supervisée, éliminant ainsi les besoins de développer des modèles de phrases ou d'annotation de données. La clé de leur approche est de concevoir un flux analytique efficace pour interagir avec le LLM et de développer des invites efficaces pour extraire les connaissances d'API du LLM et pour faire l'inférence de relations d'API. Cependant, une difficulté de cette approche est le potentiel du LLM de générer des réponses incorrectes aux demandes de relations d'API, surtout si les descriptions explicites des relations d'API existent mais sont relativement rares.

Pour remédier à cela, ils n'assument pas que le LLM peut directement répondre aux requêtes sur les relations API, par exemple "Quelle est la relation entre API1 et API2?" ou "API1 et API2 sont-ils similaires dans leur fonctionnement?". Au lieu de cela, ils emploient la méthode Chain of Thought (CoT) pour inférer les relations API, qui implique trois étapes : l'analyse de l'API FQN, l'extraction des connaissances de l'API et la décision de la relation de l'API.

Cependant, la méthode CoT met toute la description de la tâche dans une seule invite, ce qui peut conduire à une invite "épique" avec trop de responsabilités et d'accumulation d'erreurs. Pour y remédier, ils divisent la CoT en une chaîne d'IA avec trois modules : l'analyseur API FQN, l'extracteur de connaissances API et le décideur de relation API, chacun contenant plusieurs unités d'IA. Ces unités d'IA interagissent avec les LLMs étape par étape pour inférer les relations API.

L'équipe a systématiquement mené des expériences et analysé les performances de leur approche (Inference de relations API, abrégée en APIRI). Tout d'abord, ils vérifient l'utilité de deux modules importants (API FQN Parser et API Knowledge Extractor) en mesurant leur précision, qui est de 0.81 et de 0.83, respectivement. Ensuite, ils comparent avec la méthode la plus avancée (AERJE) dans trois scénarios, et les résultats montrent que le score F1 moyen de APIRI est de 0.76, nettement supérieur au score F1 moyen de 0.40 obtenu par AERJE.

Enfin, ils testent l'efficacité de la stratégie proposée dans leur approche. Les résultats montrent que la chaîne d'IA améliore la fiabilité de l'inférence de relation d'API de 67% par rapport à la méthode basée sur CoT (Chain of Thought), et la stratégie d'intelligence de foule d'IA améliore encore la robustesse de leur approche de 26% par rapport à ne pas adopter cette stratégie.

Dans l'avenir, l'équipe envisage de rendre la base de connaissances neurale Internet-scale (LLM) encore plus robuste et de renforcer l'efficacité de l'inférence de relations d'API. De plus, ils prévoient d'étendre leur approche à d'autres tâches en génie logiciel nécessitant une inferfence de connaissances.
