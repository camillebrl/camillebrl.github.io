Here is a simple tutorial, accessible to everyone (even with 0 knowledge about deep learning & LLMs), on Fine-Tuning Causal Language Models.

*Causal LLMs are the typical LLMs you heard about (gpt-type, llama-type, falcon-type, claude-type, etc), as opposed to Masked Language Models of Bert-type and Seq-to-Seq language models of T5-type*.

# Introduction
Causal Language Models are trained using 3 steps:
![](trainings.jpg)

1. **Pretraining** (Self-Supervised Fine-Tuning) : The pre-training process is the initial and most computationally expensive step of an LLM: Beginning with a randomly-initialized LLM, LLMs are trained, using a Next-Token prediction objective (we will see this later in this post), over a massive corpus of raw text. To get the best results, some balance between the model size and amount of data (row text) are determined ([Chinchilla theory](https://cameronrwolfe.substack.com/p/modern-llms-mt-nlg-chinchilla-gopher#%C2%A7training-compute-optimal-llms)). 
2. **Supervised Fine-Tuning** (usually, Instruction-Tuning): Supervised Fine-Tuning is the process of refining a pre-trained model on a specific dataset with labeled examples, allowing the model to learn from the annotations and better perform on a particular task.
3. Alignment (Reinforcement Learning from Human Feedback): Alignment refers to the process of fine-tuning an LLM to better align with the desires of human users. This is done through reinforcement learning from human feedback.

The ["base" models](meta-llama/Llama-2-7b-hf) we can find on Huggingface Hub are the models which went through the first step only (pretraining), while the ["chat" models](meta-llama/Llama-2-7b-chat-hf) we can find are the models which went through the 3 steps.

When Fine-Tuning a LLM, we need to **wonder on which step of the model we want to focus on**: If we want to modify the whole model's behavior (to give more probabilities to specific tokens, to incorporate / change knowledges into the model) we might take a "base" model found on the internet and then self-supervised fine-tune it. Whereas if we want to adapt the model to a specific task, we might take either a "base" or a "chat" model and then fine-tune it on this task. All in all, depending on what we want to do, we do not operate the same way.

# Self-Supervised Fine-Tuning
Doing Self-Supervised Fine-Tuning simply means that we **retrain the base-model on new row texts**. It can be useful to give more probabilities to specific (like technical, uncommon) tokens, or even to incorporate / change knowledges into the model. 

For Causal Language Base Models, they are initially trained by **predicting the next token of a sequence**. It can be represented like that:

![](general_idea.png)

The "loss" indicated at each prediction point represents the model's error in predicting the next token. The model's weights are adjusted through backpropagation, where the gradient of the loss function is computed, and the gradient descent algorithm is applied to minimize this loss. 
The black arrows indicate the visibility for a token with the other tokens of the sequence to predict the next token.

We will go more in detail on how this Self-Supervised Fine-Tuning works and is implemented.

## Preprocessing of the corpus (of row texts) for Self-Supervised Fine-Tuning

Each **Language Model is associated with a tokenizer**; it has its own tokenizer. This is how row text is converted into sequences comprehensible by the model (generally, sequences of discrete words / subwords). Each tokenizer has a set of tokens that can be created from a textual sequence (fixed-sized vocabulary). There are several ways of training tokenizers described in this [blog](https://vaclavkosar.com/ml/Tokenization-in-Machine-Learning-Explained). Note that each model has been trained using a specific tokenizer, so you cannot modify this tokenizer, otherwise the model will be perturbated (each token in the tokenizer’s vocabulary has a specific embedding for the model).

As example, I have a text of 30387 words in my corpus: `text = "Hallucination is a well-known phenomenon in text generated by large language models (LLMs). The existence of hallucinatory responses is found in almost all application scenarios e.g., […] this not clear whether the methods developed based on our dataset will generalize well for other LLMs."`

We are using the [transformers](https://github.com/huggingface/transformers) library to work with LLMs and fine-tune them. We can load the AutoTokenizer class from it `from transformers import AutoTokenizer`

Thanks to this class, we can easily load any tokenizer from the Huggingface Hub : `tokenizer = AutoTokenizer.from_pretrained(model_id_on_huggingfacehub)`.

I can tokenize my text (meaning transform it into something comprehensible by my model) with `tokenizer(text)`. It returns a dict object which has `input_ids` and `attention_mask` as keys. We only focus on the `input_ids` in this section. Here, the `input_ids` is a list of 6942 tokens, corresponding to the encoding of our text.

However, **each model has a maximum context size it can handle** (we will see that later). For Llama2, it's maximum context size is 4096 (whereas it is 2048 for Llama1). So we need to **split the tokenized text in blocs of 4096 tokens**. 

```
tokenized_text = tokenizer(text).input_ids[0]
blocks = []
for i in range(0, len(tokenized_text), 4096):
    blocks.append(tokenized_text[i:i + 4096])
```

Here, I get 2 blocks of respectively 4096 and 2846 tokens. A good practice is to have blocks of equal-size, meaning here, each block should be 4096 tokens (context size of the model). So I will add a padding to my second block to make it it 4096 tokens (addition of « end of sequence » token to the right, until reaching 4096 tokens):
```
 blocs[1] += [tokenizer.eos_token_id] * (4096-2846)
```

If I decode my blocs: `tokenizer.decode(blocs[0])` returns `"<s> Hallucination is a well-known phenomenon in text generated by large language models (LLMs). The existence of hallucinatory […] report the percentage of"` and `tokenizer.decode(blocs[1])`returns `"Hallucinated_ examples for each method in Table 3 on the complete DelucionQA dataset […] </s></s></s></s>"`
Note that <s> and </s> are respectively the token used in Llama2 for beginning and end of sentence. We have here several end of sentence tokens for padding our second bloc which was too small otherwise.

Now we have 2 blocs of 4096 tokens. We are ready to do self-supervised fine-tuning of the model on this corpus!

## Modelisation of Self-Supervised Fine-Tuning

To go deeper on how a Causal LM process an input, let's have a look at this schema:

![](causal_llm.png)

The input first goes through an embedding layer. The embedding layer has a unique and learnable vector embedding for every token within the tokenizer’s vocabulary. Going through this layer, each token is converted into a vector embedding.

Then, we add a position embedding to each token embedding, thus injecting positional info into the embedding of each token in the sequence. This step is crucial because it provides the model with information about the sequence order of the tokens.

The main part of the Causal LM is after this step: The combined embeddings (token & position) then enter the decoder-only transformer architecture. It starts with a Multi-Head Masked Self-Attention. 

Self-Attention is a mechanism which associates tokens with each other. In simple words, self-attention works by saying how similar each word is to all other words of the sentence, including itself. By comparing each token of the sequence with each other, it modifies the corresponding output sequence positions.

In Self-Attention, Key and Query vectors are used to compute attention weights (w) for each valid pair of tokens. These attention weights (after softmax is applied) are used to take a weighted average of value vectors, which is the output of self-attention, as shown here:

![](self_attention.png)

After the attention mechanism, the output goes through a feed-forward neural network. It usually consists of two linear transformations with a ReLU activation in between.

Finally, it results in an output token embedding, which is the learned representation of aach token after it has been processed by the model: it represents each token in the sequence's context.

Then comes the next-token prediction phase: 

![](next_token_prediction.png)

We pass each output token vector of the sequence to a Linear Layer and to a Softmax function to get the probability distribution of each next potential token of the output token vector. The token embedding of the most probable next token predicted is compared to the token embedding of the real next token as shown in the following figure. The difference between the 2 is computed through the [Cross-Entropy Loss](https://towardsdatascience.com/cross-entropy-loss-function-f38c4ec8643e).

Then, thanks to these loss, the **model is updated thanks to the Backpropagation process**. Let's go deeper in this process:
- In our pretraining step of the data, we have seen that we have 2 blocks of 4096 tokens. **Each block is an input sequence for our model** (like we had for `"<s> The cat eats the mouse. </s>"`, but here a sequence is way longer (4096 tokens)).
- **A batch is a certain amount of blocks of 4096 token in our example**. If the batch size is 64, it means that a batch is composed of 64 blocks.
- When each block goes to the network to predict each token's next token, a loss is computed. This is what we call the **"Forward Process"**.
- When the 64 blocks of a batch have passed to the network, all the loss obtained are then **aggregated to produce a mean loss for the batch**, which gives a single scalar value representing the average performance of the model on that batch. 
- Once the aggregate loss is computed, the **'Backpropagation Process'** begins. This involves **computing the gradient of the loss function with respect to each weight in the network**. The gradients indicate **how much and in which direction the weights should be adjusted to minimize the loss**. This computation proceeds in reverse order from the output layer back to the input layer, using the chain rule of calculus.
- After the gradients are computed, the weights are updated using an optimization algorithm like Gradient Descent. The basic idea is to subtract a portion of the gradient from the weights. **This portion is determined by the learning rate** - a hyperparameter that **controls how big a step we take during optimization**.

### Impact of the batch size
So here, keep in mind that the **batch size is crutial**, since it **controls after how many examples the weights of the models are updated**. A **larger batch size means that the gradient is computed as an average over more examples, which can result in a more stable and reliable direction for the weight update step**. However, a larger batch sizes **require more memory**: indeed, the loss of each example (and of each next-token prediction of each example) must be kept in memory before computing the gradient!

Note that there are some **techniques to keep this stability in the direction of the weight update without requiring too much more memories**, one of them is the **gradient_accumulation_steps**: Instead of updating the model weights after each batch, with a gradient_accumulation_steps, the **gradients from several batches are accumulated, and a single update is performed after a specified number of forward/backward passes (after the amount of gradient_accumulation_steps)**. So this mean that the loss are not kept in memory, since the gradient is anyway computed at each batch (lower here). But the model weights are updated less frequently, which result in a more stable and reliable direction for the weight update. So all in all, the **amount of examples going thorugh the forward process before the weight update can be computed with `batch_size * gradient_accumulation_steps`**.

### Impact of the Learning Rate & Learning Rate Scheduler
The Learning Rate is crutial since it **controls the step size during gradient descent** - that is, how much the weights of the model are updated during each iteration.

It is important to note that during training, it might be interesting to make this learning rate varying: 
- At the beginning of training, the model's weights are usually initialized randomly. The loss landscape at this point is relatively unexplored, and the model can afford to take larger steps (higher learning rates) to make significant progress quickly (since the loss is very high, and lot of modifications of the weights of the model might be done).
- As training progresses and the model starts to converge to a solution, the loss landscape typically becomes more complex with narrower minima (minimizing the loss). Larger steps could overshoot these minima, so a smaller learning rate helps the model to fine-tune its weights more delicately and settle into a minimum.

Towards the **end of pretraining, the model has usually learned the general features of the dataset and only needs to make small adjustments to minimize the loss**. A smaller learning rate ensures these adjustments are gradual and precise, preventing the weights from diverging due to large updates.

So it might be interesting to **make the learning rate evolve during training**. This is where the **Learning Rate Scheduler**'s role is:
- Cosine Scheduler: This scheduler type adjusts the learning rate following a cosine curve. It starts with a larger value and gradually decreases it in a cosine wave pattern to a minimum value towards the end of training. This allows for a warm-up period with higher learning rates to make rapid progress, followed by a period of fine-tuning with smaller learning rate values.
- Constant Scheduler: As the name suggests, this scheduler keeps the learning rate constant throughout training. This is simpler but doesn't take advantage of the potential benefits of adjusting the learning rate as training progresses.

For **training a model from scratch, since many weight correction might be done at the beginning, starting with a strong Learning Rate and having a cosine scheduler might be a good idea**. For fine-tuning, it is a little different: Indeed, keep in mind that when you fine-tune the model, this model might already have been trained optimally, meaning that the weights of the model have been adjusted the best way to correct the loss. Meaning that if you fine-tune it with a too large learning rate, it might modify the weights too much at each step, although you are **just adjusting it**! So you should modify it slowly. All in all, you should chose a **very small Learning Rate for fine-tuning**. A typical Learning Rate for Fine-Tuning is `2e-5` or even less.

### Impact of the amount of epochs
An epoch occurs whenever the model has been through the whole data. If `epoch=2`, the model goes through the whole data twice. If we do not have many data, it might be interesting to increase the amount of epochs.

### How to Fine-Tune a model easily
The [Trainer](https://huggingface.co/docs/transformers/main_classes/trainer) from the [transformers](https://huggingface.co/docs/transformers/index) library of [Hugging Face](https://huggingface.co/) is an interface that provides a simple way to train or fine-tune language models.

The Trainer is a class that carries out the training process using the standard training loop, which includes the Forward process, the Loss calculation, Backpropagation for computing gradients and the updating of the model's weights. It manages the automatic saving of the best models based on performance on the validation set, as well as the loading of checkpoints to resume training or perform inferences and is compatible with PyTorch and TensorFlow models.
This makes the training for Self-Supervised Fine-Tuning very simple:
```
from transformers import AutoModelForCausalLM AutoTokenizer,Trainer,TrainingArguments,default_data_collator
model = AutoModelForCausalLM.from_pretrained(model_id_on_huggingface) 
tokenizer = AutoTokenizer.from_pretrained(model_id_on_huggingface)
training_args = TrainingArguments(
    per_device_train_batch_size=1,
    gradient_accumulation_steps=64,
    num_train_epochs=3,
    learning_rate=2e-5,
    save_strategy="epoch", # the model is saved at the end of each epoch
    output_dir="fine-tuned_model", # where the model predictions and checkpoints will be written
    lr_scheduler_type='cosine'
    )
trainer = Trainer(
    model=model,
    train_dataset=dataset, # here, the dataset must be preprocessed as we have seen before
    args=training_args,
    tokenizer=tokenizer,
    data_collator=default_data_collator
)
trainer.train()
```
